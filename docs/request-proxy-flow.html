Handshake

One request, seven transitions

The runtime does not treat Kafka as a byte pipe. Each session is admitted in phases: local SOCKS5 negotiation, remote service dial, an explicit OpenAck, then a Ready signal that releases the return path.

Topic Kafka fabric ordered by key
Node B ssh dial + pump
Service 127.0.0.1:22 target socket

Active stage

Ingress through ProxyCommand or SOCKS5

A local process opens the node listener. If SSH uses tcp-over-kafka proxy, stdio is already bridged into a SOCKS5 CONNECT request.

Source path: cmd/tcp-over-kafka/commands.go -> pkg/sshproxy/sshproxy.go -> pkg/socks5/socks5.go

01

Ingress through ProxyCommand or plain SOCKS5

OpenSSH can start at tcp-over-kafka proxy --socks 127.0.0.1:12345 10.253.15.168 22. That helper dials the local SOCKS5 listener, requests the target, then relays stdin and stdout into the socket. A SOCKS5-aware client can skip the helper and connect directly.

Kafka is not involved yet. The node is still in the local negotiation phase, deciding whether it can accept the target and map it to a known remote endpoint.

02

Target accepted, route resolved, identities fixed

socks5.Accept returns the requested host:port. The node checks its routes map and turns that address into a peer tuple such as 10.253.15.168/ssh.

The outbound source is always platformID/proxy. A new random connectionID is generated here, and that identity stays attached to every frame for the rest of the TCP session.

03

KindOpen enters Kafka as the session opener

The first frame carries source platform and device, destination platform and device, and the connection ID. The Kafka message key is the stable conversation key derived from both endpoints plus that ID.

Because the writer uses hash balancing on that key, all frames for the same logical conversation stay ordered together even though the whole deployment shares one topic.

04

Peer admission happens after topic fan-out

Each node consumes the topic with a consumer group derived from its own platformID. Frames addressed to some other platform are ignored. The matching node resolves the destination deviceID through its services map and dials the local target, for example ssh -> 127.0.0.1:22.

If that dial succeeds, the peer registers an inbound session and sends KindOpenAck back to the original proxy endpoint. If service lookup or dial fails, the peer returns KindError instead.

05

OpenAck and Ready complete the cross-node handshake

When the origin receives KindOpenAck, it writes the SOCKS5 success reply to the local client and only then emits KindReady. That means the user process is told "connected" only after the peer has already opened the target socket.

The destination waits for KindReady before it starts reading bytes from the local service and publishing them back into Kafka. This avoids a race where the peer could speak before the SOCKS5 side is released.

06

Bidirectional payload relay starts after the gate opens

The outbound side chunks local reads into KindData frames up to maxFrameSize, which defaults to 32768 bytes. The inbound side mirrors the same logic in reverse, reading from the target socket and publishing return traffic back to the origin.

On receive, the runtime looks up either the outbound or inbound session registry and writes the frame payload into the corresponding TCP connection. Kafka carries bytes, but the node runtime decides which local socket they belong to.

07

Teardown is explicit, not inferred

If a route is missing, the request fails before any Kafka frame is sent and the client sees a SOCKS general failure. If the peer cannot resolve or dial the requested service, it sends KindError back to the origin.

Normal stream shutdown on either side publishes KindClose. The registries remove the session and close the local socket so the logical conversation ends cleanly on both nodes.

Wire layer

What actually rides on the topic

The wire schema is binary and versioned. Runtime state comes from frame kind, endpoint identity, and connection ID, not from a hidden out-of-band control channel.

Identity

platformID names a node. deviceID names one service on that node. Outbound sessions always originate from the fixed device proxy, which keeps reply routing deterministic.

Ordering

The runtime derives one stable conversation key from both endpoints and the connection ID, sorts the endpoint strings, and uses that value as the Kafka message key. That is the current ordering mechanism for a session.

Admission control

New readers start at LastOffset. Each node consumes the shared topic in its own consumer group and filters by DestinationPlatformID before acting on a frame.

Failure handling

Unreadable frames are discarded instead of crashing the reader. Routing and service failures return KindError. Socket shutdowns propagate as KindClose.

Topology

The request path sits inside a symmetric node deployment

The same runtime runs on both sides. Each node owns a SOCKS5 listener for outbound traffic and a service map for inbound traffic, so the flow above can happen in either direction.

The minimal shape is one broker plus two nodes. Each node subscribes to the shared topic with its own consumer group, publishes with the same frame schema, and can both originate and terminate proxied sessions.

Source map

Trace the flow in code

Read the request path from operator-facing entrypoints down into the runtime. These files are the shortest direct path through the implementation.

cmd/tcp-over-kafka/commands.go

Defines the public node and proxy commands and shows how users enter the runtime.

 pkg/sshproxy/sshproxy.go

Bridges OpenSSH ProxyCommand stdio into the local SOCKS5 connection before tunnel logic begins.

 pkg/socks5/socks5.go

Parses the SOCKS5 greeting and CONNECT request and emits the success or failure replies that the client sees.

 pkg/tunnel/client.go

Creates outbound sessions, resolves routes, sends Open, waits for OpenAck, emits Ready, and pumps local bytes into Kafka.

 pkg/tunnel/node.go

Runs the symmetric node loop, filters frames by destination platform, and dispatches open, ready, data, close, and error handling.

 pkg/tunnel/server.go

Dials the peer-side local service, sends OpenAck, and returns target bytes through serverPumpOutbound.

 pkg/tunnel/bus.go

Encodes frames, uses the conversation key as the Kafka message key, and discards unreadable frames during receive.

 pkg/frame/frame.go

Defines the versioned binary schema that every frame on the shared topic uses.
